## 一、前端技术选型

| 层面 | 选型 | 理由 |
|------|------|------|
| **框架** | **React 18** + TypeScript | 生态最大，Three.js 集成成熟（@react-three/fiber），适合3D可视化场景 |
| **构建工具** | **Vite** | 开发体验好，HMR 秒级，比 CRA/webpack 快一个量级 |
| **3D 渲染** | **Three.js** + **@react-three/fiber** | 文档指定基础3D预览（旋转/缩放/线框），Three.js 是工业标准；react-three-fiber 让 Three.js 以声明式方式在 React 中使用 |
| **3D 辅助库** | **@react-three/drei** | drei 提供了 OrbitControls、网格辅助线、环境光等开箱即用组件 |
| **UI 组件库** | **Ant Design 5** | 国内最成熟的企业级 React 组件库，Table/Form/Upload/Modal 等开箱即用，适合管理后台类产品 |
| **状态管理** | **Zustand** | 轻量、TS 友好，比 Redux 简单，比 Context 性能好，适合中型单页应用 |
| **HTTP 客户端** | **axios** + **@tanstack/react-query** | react-query 负责服务端状态缓存/轮询/重试，axios 做请求 |
| **路由** | **React Router v6** | 标准选型 |

## 二、后端技术选型

| 层面 | 选型 | 理由 |
|------|------|------|
| **Web 框架** | **FastAPI** | 原生异步、自动 OpenAPI 文档、Pydantic 数据校验、Python AI 生态无缝集成 |
| **ORM** | **SQLAlchemy 2.0** + Alembic | Python 生态标准 ORM，2.0 原生 async 支持 |
| **数据库** | **PostgreSQL 16** | 结构化数据存储（资产元数据、用户行为、缺陷数据、任务记录），支持 JSONB 灵活字段 |
| **任务队列** | **Celery** + Redis broker | 3D 生成是长耗时任务（API 调用 30s-5min），必须异步；Celery 是 Python 最成熟的任务队列 |
| **缓存/会话** | **Redis** | 文档强制要求，用于语义缓存、Celery broker、Session 管理 |
| **向量检索** | **Sentence-BERT** + Redis | 文档指定，用于相似 Prompt/修复方案语义匹配缓存 |
| **文件存储** | **本地文件系统** + 统一存储路径 | 单机部署无需 MinIO，约定 `/data/assets/` 目录结构即可，后续可平滑迁 S3 |
| **LLM 推理** | **Ollama** + Qwen3-7B-Instruct 4bit | 文档强制，注意 CPU 推理慢（预计单次 30-120s），需做好超时和排队 |
| **API 网关** | **nginx** 反向代理 | 前端静态资源 + API 代理，单机部署标配 |

## 四、开发环境与部署

| 层面 | 选型 | 理由 |
|------|------|------|
| **容器化** | Docker + Docker Compose | 统一编排 FastAPI + PostgreSQL + Redis + Ollama + Celery Worker |
| **Python 版本** | 3.11+ | FastAPI/async 支持最好的版本 |
| **Node 版本** | 20 LTS | Vite + React 18 要求 |
| **包管理** | pnpm (前端) / poetry (后端) | 各自生态最现代的包管理工具 |
| **代码质量** | ruff (后端) + biome (前端) | 速度快，替代 flake8/eslint |
| **CI/CD** | 暂不需要 | 阶段一单人单机，docker compose 本地构建即可 |

### Docker Compose 服务编排

```yaml
services:
  api:        # FastAPI 服务
  worker:     # Celery Worker (3D生成 + 后处理)
  ollama:     # Qwen3-7B 推理服务
  postgres:   # PostgreSQL 16
  redis:      # Redis 7
  nginx:      # 反向代理 + 前端静态资源
  web:        # React 开发服务器 (dev only)
```

## 五、关键技术风险与应对

| 风险点 | 说明 | 应对 |
|--------|------|------|
| **CPU 跑 Qwen3-7B 极慢** | 4bit 量化后单次推理 30-120s，用户体验差 | 做好任务队列排队 + 前端展示等待进度 + 语义缓存减少重复调用 |
| **商用API不稳定** | Tripo3D/Meshy 可能限流/宕机 | Celery 任务内置指数退避重试 + 失败告警 |
| **3D 模型预览性能** | 高面数模型在前端渲染卡顿 | 上传时自动生成低模代理（decimate），前端默认加载轻量版本 |
| **后处理耗时** | Trimesh 大模型处理可能内存溢出 | 设置面数上限 + 超时 kill + 降级方案 |

## 六、验证计划

1. **Docker Compose 一键启动全栈服务**，确认所有容器正常运行
2. **调用 Tripo3D / Meshy API**，验证文生3D、图生3D 全流程打通
3. **调用 Ollama Qwen3-7B**，验证四角色 Prompt 输出质量
4. **执行一条完整的生成→缺陷检测→修复→导出链路**，验收技术闭环
5. **前端 3D 预览器**加载生成的 GLB 模型，验证旋转/缩放/线框模式
